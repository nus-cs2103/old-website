Note that project grading is not competitive (not bell curved). Given below is the marking scheme.

Total: 50 marks

  1. Product features [max score: 15] - How good is your software as a product?

    2. Evaluated based on the live demo. Evaluated by lecturer + at least one tutor.

    ‘Half-baked’ (i.e. not production quality) features will not earn marks at all and even could be penalized. It may be better to remove such features before submission.

    • Must-have features [max score: 10]
      • -1 for each weakness
    • Extra feature
      • [score: 3 ] Reasonable implementation
      • [score: 5 ] Good implementation
  2. Design [max score: 5] - How good is your internal design? Evaluated solely based on what we can learn from the developer guide. Evaluated by lecturer.
    • Conformance with the multi-level top-down approach (i.e. design at high-level first, and do the lower-level design of each component later)
      • [score: 1] Some effort to apply top-down, but not entirely successful.
      • [score: 2] Almost entirely top-down, low coupling, high cohesion, good API design.
    • Conformance with the OO paradigm
      • [score: 1] Mostly OO, but occasional evidence of non-OO practices.
      • [score: 2] Fully OO, except in places where OO is clearly not suitable.
    • Application of patterns and principles
      • [score: 1] Reasonable attempt
  3. Implementation [max score: 10 ] - How good is your implementation? Evaluated based on an inspection of the code (submitted in your *.col files) by at least two tutors.

    4. This is graded individually.

    • Graded based on following qualities of your own code.
      • Following SLAP (IMPORTANT).
      • Following the coding standard given (IMPORTANT).
      • Using at least some assertions.
      • Using at least some logging.
      • Using at least some unit tests.
      • Commenting at the right level.
      • Exception handling.
      • Neatness and human readability.
  4. Testing [max score: 7] - How good is your testing? Evaluated based on Q&A session, developer guide, code inspection, and an independent testing sessions by the TA. You may be asked to run your unit tests during the demo.

    5. Things that may be considered: number of bugs, level of test automation

    • How much effort you have invested in the testing process
      • [score: 1] Some effort on systematic testing.
      • [score: 3] Comprehensive systematic testing with full automation.
    • How well you have tested the product
      • [score: 2] Only minor bugs.
      • [score: 4] No apparent bugs.

    Expectations :

    • Everyone is expected to write some code and write *some* tests. However, there is no need to write tests for all the code you write. This is to reduce your workload. It is sufficient if you write enough tests to prove your competency in testing. Note that in a production environment you are often required to have at least 90% of the code covered by tests. In this project, it can be much less.
    • How much of each type of testing should you do? We expect you to decide. You learned different types of testing and what they try to achieve. Based on that, you should decide how much of each type is required. Similarly, you can decide to what extent you want to automate tests, depending on the benefits and the effort required.
    • Applying TDD is optional. If you plan to test something, it is better to apply TDD because TDD ensures that you write functional code in a testable way. If you do it the normal way, you often find that it is hard to test the functional code because the code has low testability.
  5. Documentation [max score: 8 ]- How well does your user guide and developer guide achieve their objectives? Evaluated based on the user guide and developer guide. Evaluated by lecturer and TA.
    • User guide [max score: 3]
      • [score: 1] Barely good enough to get started.
      • [score: 3] Top notch user guide, both as a quick start guide and a user manual.
    • Developer guide [max score: 5]
      • [score: 1] Barely enough to get started.
      • [score: 3] Reasonable documentation, having most of these features
      • Design described in two levels or more (opposite: single class diagram, only architecture diagram).
      • Standard UML notation used in diagrams (opposite: invented your own notation, used ad hoc diagrams when there are UML alternatives).
      • Diagrams and text complement each other (opposite: separate sections for different types of diagrams).
      • Not too short or not too long. (opposite: too many unimportant stuff included, critical information missing).
      • In the specified format (page header, front page etc.)
      • [score: 5] Top notch developer guide. All above qualities are present. Neatly done.
  6. Project management [max score: 5] - How well you followed the a systematic process to iteratively evolve the product, and schedule/track project progress, and spread the work uniformly across the semester with good buffers. We look for,
    • Good version control, based on the repo.
    • Good task definition, assignment and tracking, based on the issue tracker.
    • Good use of buffers (opposite: everything at the last minute)
    • Iterative (opposite: fake iterative, doing most of the work in one iteration)

    Grading:

    • [score: 2] Two of the above.
    • [score: 5] All of the above.

Penalty for late submission: -1 mark for each hour delayed. Even a 1-second delay is penalized, irrespective of the reason. Based on the time shown by IVLE.
