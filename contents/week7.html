Pre-Tutorial Learning Activities

[team, 3pts] Move project past the 'Walking skeleton' stage

Move the code at least to the ‘walking skeleton’ stage. That is, we expect to see ‘some’ code for your project. However, it may only do something as trivial as accept a command, store it in the storage file, and display a dummy response. The important thing is it compiles, runs, and contains all the major components of the product even if the components are skeletal. It is preferred that all major components take part in the minimal functionality the walking skeleton has. It is also OK if the whole skeleton was written by one person.

Java teams: If this is your first software project, we recommend that you use the collate tool code as the starting point of your project because it is quite similar to your project (but simpler). You may also take code from the HubTurboH project, but is more complicated than your project.

[individual, 2pts] Do an OOP design for your component

In a previous tutorial, you designed the architecture for your product. As we are following a top-down approach, the next step is to design each of the components in the architecture. In other words, now that we know what the ‘big boxes’ in your design are, we need to figure out what are inside those big boxes.

Design (or refine previous design) of your own component using OO. This design need not match the current code. The code can be refactored to match the design later. If you haven’t done it already, define the detailed API for main components. Things to note:

  • The design of each component should be a separate diagram. Do not draw a single diagram with all the classes of the entire product in it.
    Rationale: We are training you to build big systems where one diagram cannot contain all details of the entire design. You need to learn how to design one component at a time.
  • The design of a component is a class diagram that show the structure of the component. It can be supplemented using object diagrams (to show object structures at various points at run time) and sequence diagrams (to show how objects interact within the component). Do not use other non-standard diagrams/notations.

When coming to the tutorial, bring the class diagram (and object diagrams and sequence diagrams, if any) for your component design. Hand-drawn rough sketches are acceptable. If you component has only one class, bring the precise API definition of that component. If you are in charge of a UI component and it has only one class, bring the definition of APIs the UI is supposed to call.

[individual, 1pts] Practice with the remote repo

Start using GitHub repo. Each member should push code to the team repo at least once. Don’t worry about messing up the repo. This is just for practice. You can reset/overwrite the repo if it gets messed up.

[individual] Update issue tracker with all the project tasks you have done so far

This is for you to practice defining tasks and recording them for better visibility. Also enter tasks that you plan to do in the next week. This plan can be revised during the tutorial. Remember to ‘close’ completed issues and milestones.

If you have many ‘ongoing’ issues but no ‘completed’ issues for the past week, that means your issues are too big. Break them into smaller parts. Reminder: read Appendix E of the handbook which gives guidelines for using issue tracker to track project work.

[individual, 1pt] Explain component design

[Please refrain from implementation-related work (coding, debugging, integrating, pulling/pushing etc.) during this tutorial. This tutorial time should be spent on design and project planning. You may do implementation-related things only after you have completed all tutorial activities.

Draw your architecture and component designs (one diagram for each component) on the board and explain them to the team and the tutor. Refine the design if necessary.]

We ask you to draw the diagrams again (even though you have brought diagrams drawn at home) to give you more practice in on-the-spot hand drawing of minimalistic diagrams to aid discussion, which is different from drawing detailed diagrams to document the design. When drawing quick diagrams for discussion, avoid adding unnecessary details. For example, you can omit properties and methods in the class diagram, unless it is an API exposed to other components. Refer the digram below for an example.

Do not draw a one giant diagram with all the details. Instead, draw separate diagrams for each component, as shown above.

When explaining, do not try to explain the program flow using class diagrams. Class diagrams are structure diagrams, not behavior diagrams. If you want to explain program flow, you should use sequence diagrams.

Resources:

Part 3 - Putting up defenses to protect our code [40 minutes]

Slides

Overview: Proper error handling is a must for any production system. The defensive coding approach can lower the risk of component failure even further.

Topics: error handling, defensive programming

Resources:

  • Assertions: At least a few assertions are expected by V0.3.
    • Java: Programming With Assertions (a guide from Oracle).
    • C++: assert (from MSDN).
  • Exceptions: You are required to use exceptions properly by V0.3.
    • Java: Exceptions (a tutorial from Oracle).
    • C++: Exception handling (from MSDN), A four-minute introduction to C++ exceptions (Youtube video) , Another tutorial written by a developer
  • Logging: You are required to do *some* logging by V0.3. However, it is enough to use normal file I/O to write to a log file (similar to how you did file I/O in CE1/2). Using a logging frameworks is optional.
    • Java: Logging using java.util.logging (a tutorial by Jakob Jenkov), A more detailed tutorial (from Oracle).
    • C++: While there are a few C++ logging frameworks available (e.g., Boost-log), they look rather complicated. You may be better off using direct file I/O for writing to a log file.
