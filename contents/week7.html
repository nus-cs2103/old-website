T6A1. Work in a 2KLoC code base, as a team

To be done with your Phase C team members.

Set up a GitHub organization for your Phase C, as described here. No need to set up a repo for the project yet. Setting up the org is enough.

Complete the exercise in Revision Control Software: Follow the Feature Branch Workflow [LO-FeatureBranchWorkflow] but do note the following:

  • [Step 1]
    • Use the GitHub organization your created above.
    • Repo to be forked to your new organization: Address Book Level 3.
  • [Step 2]
    • You can either clone the forked repo (i.e. /yourorg/addressbook-level3) to your computer or add that repo as a 'remote' to your existing addressbook-level3 clone on your computer. (a local repo can have multiple remote repos as long as they share a common revision history).
    • Feature to add: Do the exercise in AddressBook-Level3: Work in a 2KLoC code base [LO-2KLoC], but take note of the following.
      • Discuss with team members and decide which feature each of you will implement (to avoid duplicating the same feature).
      • Name your branch as some-keywords-related-to-the-enhancement e.g. add-edit-command instead of following the usual branch naming convention for tutorial activities.

Submission: This is a team submission. Create a PR from the master branch your team repo to nus-cs2103-AY1617S1/addressbook-level3. PR name is similar to previous tutorial activities, but omit the student name. e.g. [T6A2][T11-B3]

T6A2. Get started on the project requirements and design

To be done with your Phase C team members.

Start drafting V0.0 Deliverables

Suggested sequence : document product survey --> document user stories --> prioritize use stories --> select user stories to include in your product --> design the product --> write the user guide
Fill up Glossary and NFRs in parallel to the above.

You may use a simple collaboration medium such as Google Docs for the above. You can transfer the content to your GitHub repo in week 8 after the content has become somewhat stable. Given that you will be converting the document to Markdown later, minimize effort spent on formatting.

Submission: Proper submission will be in Week 8 (after converting to markdown and pushing to the repo). In tutorial 7, just show your intermediate drafts to the tutor.

Lecture 7

Part 1 - The View from the Top: Architecture [40 minutes]

[Handout] [Slides]

Overview: The architectural design should break the product into a small number of well-specified components that can be developed in parallel. UML sequence diagrams can be used to figure out the API of those components.

Topics: software architecture, components and APIs, modeling component behavior using seq diagrams, different approaches to design

Resources:

  • [Must watch] A three-minutes video we created for you Game of Tic-Tac-Toe: A worked example of designing architecture and discovering component APIs

Part 2 - Putting up defenses to protect our code [30 minutes]

[Handout] [Slides]

Overview: Proper error handling is a must for any production system. The defensive coding approach can lower the risk of component failure even further.

Topics: assertions, logging, defensive programming

Resources:

  • Refer Address Book (Level 4) Learning Outcomes page for more resources.

Part 3 - How to Avoid a Big Bang: Integrating Software Components [15 minutes]

[Handout] [Slides]

Overview: Your integration approach play an important role in how you implement components. When integrating work done by multiple developers, revision control software can be a big help.

Topics: Integration, Build automation, Continuous Integration, Dependency Management
