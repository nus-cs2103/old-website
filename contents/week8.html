Pre-Tutorial Learning Activities

Reach V0.1 [Team, 10pts]

Reach V0.1. Refer the handbook to see the minimal requirement for V0.1. The milestone V0.1 should be reached *before* the tutorial. No pre-demo code tinkering is allowed during the tutorial.

Relate Coupling and Cohesion to your component [Individual, 2pts]

Try to relate coupling and cohesion to the current architecture or the design of your component. Where can you improve those two aspects? Think of an alternative design and try to explain the difference between your design and the alternative design using coupling and cohesion. You will be asked about this during the tutorial.
Recommended: Bring two design diagrams to the tutorial -- one representing the current design of your component and the other representing an alternative design.
If you component has only one class, you can try to come up with an alternative that has multiple classes. The alternative can be worse or better than the current design. The important thing is you can compare the two in terms of coupling and cohesion.
The diagrams should be proper UML diagrams: class diagrams (if you plan to compare the two in terms of structure) or sequence diagrams (if you plan to compare in terms of behavior).

Update the project plan in the issue tracker [Individual, 2pts]

Update the issue tracker with all the project tasks you have done so far. Also enter tasks that you plan to do in the next week. This plan can be revised during the tutorial. All issues allocated to V0.1 and prior milestones should be closed. If any are not done yet, reschedule them to future milestones. A person should typically have some closed issues (in V0.1 milestone) and some open issues (in the upcoming milestones) against his/her name at this point. Status of those issues should be updated accordingly.

Ask for help if you have fallen behind too far [Individual]

We just passed the halfway point of the module and we have been pushing you to learn things by yourself. However, we don’t want to leave anybody behind either. If you think you have fallen too far behind and feel like you cannot catch up, please let your tutor know, or email the lecturer. We’ll try to help you catch up.

Tutorial 7: V0.1 demo, coupling and cohesion

Demo V0.1 [Team]

Do a demo of your V0.1 to your tutor. It is an informal demo to show what you have done so far. This demo can be done by one person. We expect you to take turns in doing the demo. Those who did not do the demo this time are expected to demo in future weeks. This is a live demo (i.e. use the actual product for the demo).
Demo duration ~5min. No need for slides.

Explain coupling and cohesion in your component [Individual]

Explain coupling and cohesion of the design of your component.
Note: If you don’t have neat (can be hand drawn, but should be neat) diagrams for your architecture and component designs at this stage, it is a sign that you are simply hacking together code and there is no effort to follow a design.

Ask for help if needed [Team]

Inform your tutor if you there is something you cannot manage on your own and need our help.

Refine the milestone plan [Team]

Now that you have some idea about the effort it takes to implement some user stories, refine the milestone plan for V0.2-V05. i.e. which story will be delivered when. Use HubTurbo to show the milestone plan to the tutor.
Tip: Create a board called 'Milestone Plan', containing a panel for each milestone. Show that board to the tutor.

Refine Task Schedule [Individual]

Each person should refine in the Issue tracker your task schedule for the next week.
One member should use HubTurbo to show the latest work allocation to the tutor.

After the tutorial: Submit week 7 wrap up session [Individual]

From this week onwards, the weekly wrap up session on TEAMMATES contains a compulsory question. Please remember to submit it after the tutorial.

Intended Learning Outcomes

Can relate coupling and cohesion to a design.

You should be able to justify some design decisions using the concepts coupling and cohesion. e.g. We chose design 1 over design 2 for the Parser component because in design 1 the Parser does not execute commands, which makes it more cohesive.
