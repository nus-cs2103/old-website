Pre-Tutorial Learning Activities

T7A1. Set up the sample project

Only one team member:
  1. Ensure the GitHub organization name is as specified in our requirements.
    Note: GitHub allows renaming orgs/repos.
  2. Fork Address Book Level 4 to your team org.
  3. Rename the forked repo as main. This repo is to be used as the repo for your project.
  4. Clone the repo to your computer.
  5. Edit this element <name>addressbook-level4</name> and the line below that in the .project file to match the *product name* (not repo name) of your project. e.g. SuperToDo
    Commit the change.
  6. Push to the main repo you created above.
  7. Ensure your team members have the desired level of access to your team repo
All team members:
  1. Clone to your Computer the main repo your team member created in your team's GitHub org.
  2. Set it up as an Eclipse project (follow the instructions carefully).
  3. Ensure you can pass all the tests.
  4. Read the 'Design' Section of the Developer Guide to get a high-level understanding of the design before you start reading the code.

T7A2. Complete V0.0

Work as a team to complete V0.0 that you started last week. Transfer the content to the relevant pages in your main repo.
  • Refer to the 'workflow' section given at the end of our GitHub instructions to find possible workflow options.
  • MUST DO : Complete the UserGuide.md page. This page should tell us in detail what is the product you plan to deliver at the end of the semester. Remember to include detailed command formats and UI mock ups. We are going to use it to give feedback about your product design over the next week.
  • Update the AboutUs.md page. In particular, indicate which member is in charge of which component (i.e. UI, Logic, Storage, etc.).
    Note: Being 'in charge' of a component does not mean you work in that component only. Rather, you are expected to know it well and possibly review/approve changes done to that component by others.
    Delete details/photos of original developers. Include your own details and photos instead.
    Include the name and photo of the project mentor (i.e. your phase C tutor). His/her photo can be taken from the Teaching Team page.
  • Update the README.md page to match your project. In particular, add a UI mockup of your intended final product. Also, remember to acknowledge the original source of the code i.e. The sample project created by SE-EDU initiative at https://github.com/se-edu/
  • Complete the appendices of the DeveloperGuide.md within the next two weeks.
Submission:
Create a PR from your team repo master branch to https://github.com/nus-cs2103-AY1617S1/addressbook-level4. Name it [T7][Your-Team-ID] e.g. [T7][W09-C3]

T7A3. Reach V0.1

Work as a team to deliver V0.1. Note the following:
  • Aim to deliver at least add/delete/find support for floating tasks. You should be able to do this by deleting/renaming the given Address Book code, without having to add too much new code.
  • Please watch the webcast if you missed Lecture 7. We noticed that in the previous week many teams wasted time designing an unrealistic UI for their product because they did not pay attention to the sample UI shown in Lecture 6. Lecture 7 too contained some information important for V0.1.
  • Change code in small steps, run tests after each step, commit each step, and merge often.
  • Following things can be dropped if you find them too much trouble. They are not compulsory to have in the final version.
    • Automated GUI testing
    • Using Gradle to build the project
    • Using Travis for CI
    • Using regular expressions for parsing: feel free to switch to a simpler technique.
    • Using new Java 8 lambdas and streams : feel free to use normal for loops instead.
  • If you feel Level 4 code is too daunting for you to use as the starting point, you may use Level 3 code as your starting point instead.
Submission:
No separate submission required as your T7A3 work will be captured in the T7A2 submission above.
