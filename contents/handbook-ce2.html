Here are the things to do for CE2 (read all instructions before you start CE2).

  1. After CE1 code written by your classmates are made available on IVLE, do look for code that reads better than yours. Think about what makes them more readable than yours. If you did not attend lecture 2, please watch the webcast to learn what we consider ‘good code’ in this module.
  2. Refactor CE1 until it reaches a high quality standard. Remember to consider feedback received from CE1 code review. In particular,
    1. Take into account the coding standards we have published (in the schedule page).
    2. We expect to see short and simple methods that follow SLAP.
    3. Also follow other practices recommended such as good naming and the right level of commenting.

    Note that those who don’t reach the expected code quality in CE2 may have to submit CE3 as well.

  3. Add these additional functionality.
    • A sort command to sort lines alphabetically.
    • A search command to search for a word in the file and return the lines containing that word.
  4. When adding above features, do it in TDD fashion.
    5. Hints:
    • If you are new to unit testing, try adding some unit tests to existing CE1 functionality first before trying TDD. However, you are not *required* to add tests for existing CE1 functionality.
    • TDD is not simply ‘writing tests before writing functional code’. Refer the demo in lecture 4 to find out how to TDD in small steps.
    • When unit testing the ‘sort’ command, you can use existing ‘display’ and ‘add’ commands in the testing process. e.g., add a line, sort, display, and check if the display is sorted properly. This is not strict unit testing since we are not testing ‘sort’ functionality in isolation from other functionality, but it is acceptable since those other functions are also under your control and tested by you.
    • Having a public method that takes the user command and executes it (CityConnect uses this approach) can greatly simplify the testing process. Then, you can write tests like this: 
      • //check if the “clear” command returns the right status message
assertEquals(“all content deleted from mytextfile.txt”,
TextBuddy.executeCommand(“clear”));
//check if the file was actually cleared
assertEquals(0, TextBuddy.getLineCount());
  5. There is no need to test using input/output redirection method used in CE1. Instead, you should use a proper unit testing framework in CE2.
  6. Commit the code to your repo frequently, after each step in refactoring/TDD (e.g., add a test → commit → add code to pass the test → commit, and so on). Doing so will help us see the steps you followed during coding. VisualStudio users: Avoid adding binary files and temporary files to Git. These can make your repo too big. See here for a list of file types you should not add.
