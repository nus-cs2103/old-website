Pre-Tutorial Learning Activities

Read some design tips

If you haven’t already, Also look at CS2103 Project: Points to Ponder for some design tips useful when refining the product design.

Use other models [Individual, 2pts]

Do one of the following (try to ensure at least one member does each):
  • Relate (no need to draw) at least 2 “non-examinable” UML models to your project. Explain why/where the model is potentially applicable or why it is not applicable to the project.
  • Draw an activity diagram to represent a workflow/algorithm in your part of the project. If your part of the project does not have any workflows, you can draw an activity diagram to represent the workflow of a user using your team’s product.
    Bring the diagram to the tutorial.
  • Draw an OODM for your project. Remember, OODM is about the problem domain, not about the solution. Classes such as GUI, Manager, Storage should not appear in the OODM but it can have classes such as Task and Reminder.
    It should be about how a person deals with his/her schedules and tasks. Bring the diagram to the tutorial.

Do some integration/system testing [Team, 5pts]

Ensure you have *some* automated integration and/or system tests in place.
i.e. testing beyond individual unit testing and covering a major part of the system, not just one method/class.
e.g. a test that takes the full user command as the input and verify that it is executed properly.

Contribute to V0.3 [Individual, 2pts]

Commit code for this version.
Try to ensure your latest version of the product works before commiting code. i.e. do not commit code that breaks the product.
You are encouraged to add tags in the repo to identify intermediate working versions. e.g. V0.2.1. That way, you can fall back on the latest working version if the intended V0.3 was not reached by the deadline.

Reach V0.3, with time to spare [Team, 3+2=5pts]

[3pts] V0.3 should be noticeably better than the previous version, The code should be tagged as V0.3 in the repo.

[2pts] The current version should be ready for dog fooding.

[+1 bonus point] Reach V0.3 with at least 12 hours in advance. This is to encourage you to have buffers. i.e. set internal deadlines ahead of external deadlines.

Be ready to demo, with good demo data [Individual, 2+1=3pts]

[1pts] ALL members should have a demo-ready V0.3 version in your computer when you come to the tutorial.
Any of you should be able to do the demo at a moment’s notice.

[2pt] Add at least 20 realistic tasks in the demo database covering a wide variety of tasks. Even better if you use the product to track your own schedule for real.

Project planning [Individual, 2pts]

As before. All V0.3 issues must be closed and each person must have some issued allocated to them for V0.4.

Revisit marking scheme

This may be a good time to revisit the V0.5 marking scheme given in the Handbook while you still have time to tweak things to get more marks.

Tutorial 10: More models

Demo V0.3

Every member must show their demo-ready version of the product containing demo data. Your tutor will ask one of you to do the demo.

Show results of pre-tutorial learning activities to tutor

Show models, integration/system tests, revision history, and the issue tracker to the tutor.

After the tutorial

Dogfood your product (i.e. use it for your own task tracking) intensively and use that experience to improve it as much as you can during the next 1.5 weeks.

Intended Learning Outcomes

Knows about UML models. Can use some of them.

Should be able to use use case diagrams, sequence diagrams, class diagrams, object diagrams, activity diagrams, OO domain models.
Should be aware of remaining UML models. e.g. state machine diagrams, package diagrams, component diagrams, etc.

Can automate integration testing.

Based on the relevant pre-tutorial activity.
