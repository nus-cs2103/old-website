[Fri 23:59] CE1: Code Review

[individual, 5x2 pts]

In this exercise, you will review the code of two other classmates.

Rationale: Code reviews are an important part of a software engineer’s life. While some of you may not be experienced enough to be a good code reviewer, doing a code review is still a valuable opportunity to experience code from the reader’s point of view. It will also help you to improve the quality of your own code.

Procedure:
  1. Check the review assignment (i.e. who will review whose code) here.
  2. Download the pdf files submitted by the two assigned reviewees, from the IVLE workbin. Note that some may not have followed the submission instructions closely and named their files in different ways. Do a Ctrl+F in the workbin page to find the files submitted by the reviewee. Some may even have put the pdf file inside the zip file. Yes, it is quite annoying when people do not follow the instructions :-p
  3. Refresh your memory about the coding standard you were expected to follow. Links to the standard are given in the Tutorial 2 description above.
  4. Review the code for the following aspects. Note that we are reviewing the coding style only, not the accuracy or the performance.

    Formatting:
    a) Code formatted with syntax coloring to help the reader.
    {opposite: Code just dumped into a document without much regard to the readability. Some statements overflow to the next line}

    Names - meaning:
    b) Class/method/variable names are meaningful and intuitive.
    c) Boolean variables or methods returning boolean values are named to indicate their boolean nature e.g. isValidName hasExtraSpaces()
    d) Variable names are based on nouns (e.g. maxValue) and method names are based on verbs (e.g. getValue)

    Names - case:
    e) Class/method/variable/constant names use upper/lower/Camel/Pascal case appropriately e.g class name: CityConnect (Pascal case), method name : getValue (Camel case), constant name MAX_VALUE (upper case with underscores)

    Comments:
    f) Comments are used to explain WHAT and WHY aspects of the code, not HOW aspects.
    g) No trivial comments.
    h) Minimal but sufficient level of commenting.

    Magic numbers
    i) No magic numbers/strings.

    Layout:
    j) Code neatly laid out to improve readability.
    k) Braces placement, indentation, and alignment follow the standard.
    l) Braces are used even for blocks with just one statement.
    e.g. if(isValid){
    processInput(input);
    }

    SLAP:
    m) Short methods with not too-deep nesting.
    n) Each method written generally at a similar level of abstraction.

    Testing:
    o) A reasonable set of test cases to cover various usage scenarios.

  5. Submit your findings using the TEAMMATES session ‘CE1 code review’ (open from Week 4 Mon, 7pm). Aspect above for which the code needs improvement, give examples when you submit the review.

    e.g.
    a) The variable name t2 is not very meaningful
    k) Braces in the getLines() method do not follow the standard.
    m) Not enough SLAP in the main method. It can be shorter and the nesting can be reduced.

  6. A code review requires both technical skills and people skills. Be honest in the review. Your review will not affect the reviewee’s CE1 grade.
    Be kind too. There is no need to criticise harshly. No need to be witty. Just offer gentle suggestions on how to improve.
  7. If it seems the reviewee’s has not submitted CE1, review the code from another classmate (of your own choosing).

If you would like to get a head start on coding your project, you can do coding experiments around this time to figure out how to implement certain functionalities that you might need later in the project. Such solutions are called spike solutions. Here are some examples for which you can create spike solutions.

  • Find a way to change color in text output.
  • Create a simple GUI that has a text box for input and a textarea for output.
  • Explore suitable data formats (e.g json, icalendar, xml, etc.) for saving data for your project. Find libraries for easy writing/reading of data in the selected formats.

Lecture 4: Let's split work and start building this thing

Pre-lecture activity

Give this problem a try (at least some thoughts) before coming to the lecture. We are going to do it as an exercise during the lecture (30 minutes).

Imagine you are building an online system called TeamFormer. TeamFormer is a system for helping CS2103 students during team forming period. It lets students record who is in which team. Students can use the system to find teams short of members and students looking for teams, easing the team forming process. Note that the system is not meant as a platform for students to communicate with each other to find team members. Students are supposed to talk to each other and form teams outside the system, as they do now. TeamFormer simply helps students to see the team forming status of the class (e.g., who has teams and who don’t) and record their own team after forming a team. It is also meant to ease the team-forming-related workload of instructors.

What should be the following elements of TeamFormer requirements?
  • Vision: {A short statement describing the product’s objective/direction}
  • User types/roles: Students, Instructors, {any other user types?}
  • High-level workflow: {In the process of team forming, who will do what first, and what will happen next, and so on? Where does TeamFormer fits into this workflow?}
  • Scope: {What will TeamFormer do, and more importantly, what team-forming-related things it will NOT do?}
Specify important requirements of TeamFormer as a list of User Stories.
Write one use case (i.e., textual form) for one of the user stories that you think will result in the most complex user interaction.

[35 minutes] Part 2 - Never Too Early to Test: an Introduction to Early Developer Testing

Slides

Overview: Testing should be done earlier and at lower levels without relying entirely on system testing or higher-level component testing. API testing can be automated using testing frameworks such as JUnit, and with the help of drivers and stubs. TDD is a unique approach to ensure the testability of the component from the beginning and not as an afterthought.
