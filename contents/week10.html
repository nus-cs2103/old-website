Pre-Tutorial Learning Activities

Reach V0.2 [Team, 5pts]

This should be a working version that is better than the previous version in some way. If required, update the project manual. For these intermediate milestones, the amount of improvements between versions is not important. What matters is you moved the product forward and everyone played a role in doing so. If your team is busy with other modules, you can chose to do less (but you should do some), improvements. But doing very little up to V0.4 and doing a lot in V0.5 is not acceptable.
Remember to git tag your V0.1 and V0.2 in the repo.

There should not be any pushing/pulling or tinkering with code during the tutorial. You are expected to have a stable V0.2 version in the repo before you come in. This V0.2 may or may not be the latest code in the repo.

Grading:
  • 2 points will be given for having a proper release. i.e. tagged as V0.2 in the repo, tagged version loaded into demo computer, and ready to start the demo.
  • 3 points will be given based on the progress of the product. Those that are ready to start dog fooding will earn an extra bonus point.

Do some systematic unit testing [Individual, 2.5+2.5=5pts]

(a) [2.5 pts] Ensure at least some automated tests exist for your part of the code.
Note: If you are doing the GUI component, you might find it harder to do unit testing in there. In that case, you can get some more work assigned to you (e.g., writing a utility class) from somewhere else for which you can write unit tests. The important thing is to show us that you can write unit unit tests. Locating a place to do that is up to you.

(b) [2.5 pts] Insert at least some comments in the code to explain the relation between your test cases and the test case design heuristics covered in the lecture.
e.g., /* This is a boundary case for the ‘negative value’ partition */
Note: For some SUTs, it is not easy to identify partitions and boundaries. Try harder. If still cannot, look elsewhere. The important thing is to show us that you can apply test case design heuristics in practice. Locating a good example to do that is up to you.
During the tutorial you will be asked questions such as the below. Please read the relevant sections of the module handout to beef up your knowledge about test case design heuristics.
  • What is an equivalence partition?
  • How does it help to improve E&E of testing?
  • For a selected test case: What partition does this test input belong to? What is another input from the same partition?
  • What is a boundary value?
  • and so on...

Update the issue tracker [Individual, 2pts]

Update the issue tracker to reflect what you have done for the past week and what you plan to do in the next week.

After the tutorial

  • Start dog fooding: If your V0.2 is reasonably feature-complete, start using it to manage your own schedule. If it is not complete enough for dog fooding, revise your project plan so that you can start dog fooding at least by V0.3.
  • You are encouraged to look at V0.1 project manuals of other teams. It is OK to adopt good ideas from others, as long as you give credit to them.
  • Also look at CS2103 Project: Points to Ponder for some design tips useful when refining the product design.
