Tutorial 5: Unit testing, project planning, task tracking

[Team, 2pts] Define project tasks.

  • Re-schedule user stories among milestones, if necessary.
    You may wish to schedule 'risky' features earlier so that you have enough time to recover if their implementation did not go according to expectations.
  • Assign components in your architecture among team members. i.e. who will implement which component. If you have fewer components than team members, consider dividing bigger components into sub components.

    Note: See if this tip can improve your design.
  • Decide projects tasks that needs to be done in the next week, assign them to team members. The allocation of tasks (i.e. who does which task) and task status labels (status.done etc.) should be updated at least weekly.
    You may wish to break bigger tasks into smaller manageable size. For example, you can designate a big user story as type.epic and create several smaller user stories to represent parts of that epic. Similarly, you can create smaller tasks to represent user stories.
    Must read: See Appendix E of the handbook Guidelines for using issue tracker to track project work.
    Some example tasks for 'User can add a floating tasks' user story:
    • UI support for adding floating tasks (assignee: David, milestone V1.0)
    • Parser support for adding floating tasks (assignee: Chin Chin, milestone V1.0)
    • Controller support for adding floating tasks (assignee: Ravi, milestone V1.0)
    • Storage support for adding floating tasks (assignee: Jean, milestone V1.0)
    • Data support for adding floating tasks (assignee: Jean, milestone V1.0)
