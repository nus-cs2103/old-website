Pre-Tutorial Learning Activities

Use HubTurbo

Start using HubTurbo to access GitHub when it helps. Make a note of which features of HubTurbo is done well and which are not done well. These observations can help you in designing your own product. Keep in mind the intended audience of HubTurbo is different from your own project, so be careful when adapting ideas from HubTurbo to your project. Feel free to report bugs and suggest improvements in the HubTurbo issue tracker.

Watch this 3 minutes video
Game of Tic-Tac-Toe: A worked example of designing architecture and discovering component APIs.

[Individual, 3 pts] Propose an Architecture for your product

Propose an architecture for your product. i.e. Divide the product into a small number of components. Discover some of its component APIs. Where applicable, draw some sequence diagrams (at least one) to support the discovery of APIs. Keep the hard/soft copies of your diagrams to bring to the tutorial (rough hand-drawn sketches are OK too). If you are not very sure about how to draw sequence diagrams, you can do the following optional team exercise to increase your confidence before attempting the above task.

[Team, Optional] Propose an Architecture for text-based Minesweeper

Propose an architecture for the text-based Minesweeper game used as an example in lecture 4. Discover the API of its components using sequence diagrams.

Tutorial 4: Discovering components and APIs, architecture decision

[Team, 2pts] Draw architectures and decide on an architecture for the project

  • Draw your individual architectures (that you came up with previously) on the whiteboard. Note that an architecture need to be simple enough to draw quickly from memory. Compare and contrast with team members.
  • Decide an architecture for the project. This can be one of the architectures proposed by a team member, or a new one created by combining those proposals. You are required to use the whiteboard for this exercise. To increase peer learning, leave your diagrams on the whiteboard for the next team to see.
  • For the chosen architecture, draw relevant sequence diagrams and discover some API definitions. For example, there should be an SD to illustrate how the architecture supports the use case of adding a task.
    Important: This is not a low-level detailed design. The design should be kept at high-level (i.e architecture level). Do not go to the level of classes and data structures.
  • Once this is done, assign components to team members for further detailed design and implementation.
  • Show to the tutor the chosen architecture, corresponding sequence diagrams, and APIs before you leave the tutorial.
  • You are welcome to look at the diagrams of other teams before you leave the room. See how their architecture differs from yours. If you find something that is better than what you came up with, you are welcome to adopt the better approach.

After the Tutorial

Continue discovering and documenting component APIs until you are sure your APIs can support at least the commands you are going to support for V0.1

Intended Learning Outcomes

Can define an architecture for a small project, in top-down fashion.

Can derive component API systematically.

Can match product design to requirements. Aware of the need for 'good' product design.
